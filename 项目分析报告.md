# AI 模型选择器项目分析报告

## 一、项目概述

### 1.1 项目定位
这是一个 **AI 模型价格对比平台**，聚合了 25 家国内外 AI 平台的 2353+ 个模型，提供实时价格对比、智能推荐、月费计算等功能。

**核心价值**：
- 一键对比全网 AI 模型价格
- 跨平台比价（同一模型在不同平台的价格对比）
- 月费估算和预算反推
- 智能推荐最优模型

### 1.2 技术栈
- **前端**：纯 HTML + CSS + JS（零依赖，单文件架构）
- **后端**：Python 数据抓取脚本（generate.py）
- **部署**：Vercel 静态站点 + GitHub Actions 自动更新
- **数据存储**：JSON 文件（models_data.json, official_prices_db.json）

## 三、核心流程分析

### 3.1 数据获取流程

```
generate.py 运行
    │
    ├─→ 检查 models_data.json 是否存在且未过期（24小时）
    │       │
    │       ├─→ 是：从 JSON 加载数据（快速模式）
    │       │
    │       └─→ 否：从各平台 API 实时获取
    │               │
    │               ├─→ 国内平台（阿里百炼、硅基流动、月之暗面等）
    │               │       └─→ 调用 OpenAI 兼容 API 获取模型列表
    │               │
    │               ├─→ 国外平台（OpenRouter、Together AI、Groq 等）
    │               │       └─→ 调用 API 获取模型列表和价格
    │               │
    │               └─→ 聚合平台（n1n.ai、ChatAnywhere 等）
    │                       └─→ 从公开 API 或网页抓取
    │
    ├─→ 价格解析（SSOT 四层系统）
    │       │
    │       ├─→ T1: API 直接返回的价格（最高优先级）
    │       │
    │       ├─→ T2: fetch_official_prices() 官方网页爬取
    │       │
    │       ├─→ T3: official_prices_db.json 查表
    │       │
    │       └─→ T4: LiteLLM 社区价格（海外平台兜底）
    │
    ├─→ 生成模型卡片 HTML
    │
    ├─→ 写入 models_data.json
    │
    └─→ 生成 index.html
```

### 3.2 价格获取机制（SSOT 系统）

**设计理念**：单一真实来源（Single Source of Truth），确保价格可追溯、可验证。

```python
def get_absolute_price(platform, model_name, api_price=None):
    """
    严格四层 SSOT:
    1. API 直接返回的价格（T1 平台）
    2. 官方爬取结果（fetch_official_prices）
    3. official_prices_db.json（精确匹配 → 前缀匹配）
    4. LiteLLM 社区价格数据（海外平台兜底）

    找不到 → 返回 (0, 0, "N/A", "") 并打印 WARNING
    """
```

**价格来源标签**：
- `A` = API 实时
- `S` = 官方定价页爬取
- `DB` = 官方价格数据库
- `L` = LiteLLM 社区数据
- `P` = 代理平台自营价（非官方）

## 四、代码质量分析

### 4.1 优点

1. **架构清晰**：SSOT 四层价格解析系统设计合理
2. **容错性强**：多层缓存、重试机制、硬编码兜底
3. **自动化程度高**：GitHub Actions 每日自动更新
4. **零依赖前端**：纯 HTML/CSS/JS，加载快
5. **功能丰富**：筛选、排序、对比、月费计算、TTFB 测速等

### 4.2 问题识别

#### 问题 1：单文件过大（严重）
- `generate.py` 约 3000 行，包含数据获取、价格解析、HTML 生成等所有逻辑
- `index.html` 约 3MB，包含所有前端代码和模型数据

**影响**：
- 难以维护和测试
- 代码复用困难
- 新人上手成本高

#### 问题 2：API Key 硬编码（安全风险）
```python
# 第 12-34 行：API Keys 直接写在代码中
SF  = os.environ.get("SF_KEY", "sk-lvhjyumsmqidzpmwtmkcyxrhetbmaodfjklenoomnlsjbqha")
ALI = os.environ.get("ALIYUN_KEY", "sk-5521c543f8f74954a027ddd41edafa08")
# ... 更多硬编码 Key
```

**风险**：
- 代码泄露会导致 API Key 泄露
- 违反安全最佳实践

#### 问题 3：缺乏单元测试
- 没有针对核心函数的单元测试
- 难以验证价格解析逻辑的正确性
- 重构风险高

#### 问题 4：错误处理不完善
```python
# 多处使用空 except 捕获所有异常
except:
    pass
```

**影响**：
- 难以定位问题
- 可能隐藏严重错误

#### 问题 5：代码重复
- 多个平台的数据获取逻辑高度相似，但重复实现
- 价格解析逻辑在多处重复

#### 问题 6：缺乏类型注解
- Python 代码没有使用类型注解
- 降低代码可读性和 IDE 支持

#### 问题 7：前端代码内嵌
- 所有 JS/CSS 内嵌在 HTML 中
- 难以维护和调试
- 无法利用现代前端工具链

        """验证必要的 API Key 是否配置"""

    if not os.path.exists(cache_file):
        return False
    age_hours = (time.time() - os.path.getmtime(cache_file)) / 3600
    return age_hours < CACHE_EXPIRY_HOURS
